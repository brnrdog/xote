Xote is a ReScript-first library, but the package also ships JavaScript modules that can be imported by JavaScript and TypeScript applications.

The main difference is syntax: ReScript projects usually use Xote's JSX integration, while JavaScript and TypeScript projects should use the runtime constructors directly.

## Install

```bash
npm install xote
```

## Use the Client Entry

For browser-rendered UI, import from `xote/client`. It contains the core rendering modules without router, SSR, hydration, or MDX code.

```js
import { Signal, Computed, Effect, View } from "xote/client";
```

Use the other entries only when a feature needs them:

- `xote/client` for client-side views
- `xote/router` for routing
- `xote/ssr` for server rendering
- `xote/hydration` for client hydration
- `xote/mdx` for MDX integration

## JavaScript Counter

Build Xote views with `View.element(tag, attrs, events, children)`. Attributes are `View.attr(name, value)` pairs. Events are `[eventName, handler]` pairs.

```js
import { Signal, Computed, Effect, View } from "xote/client";

const count = Signal.make(0);
const doubled = Computed.make(() => Signal.get(count) * 2);

Effect.run(() => {
  console.log("Count:", Signal.get(count));
});

const increment = () => {
  Signal.update(count, n => n + 1);
};

const app = View.element("main", [View.attr("class", "counter")], [], [
  View.element("h1", [], [], [View.text("Counter")]),
  View.element("p", [], [], [
    View.text("Count: "),
    View.signalText(() => String(Signal.get(count))),
  ]),
  View.element("p", [], [], [
    View.text("Doubled: "),
    View.signalText(() => String(Signal.get(doubled))),
  ]),
  View.element(
    "button",
    [View.attr("type", "button")],
    [["click", increment]],
    [View.text("Increment")],
  ),
]);

View.mountById(app, "app");
```

The reactive reads happen inside `Computed.make`, `Effect.run`, and `View.signalText`. When `count` changes, the text nodes update in place.

## TypeScript Setup

Xote generates TypeScript declarations from the ReScript modules with `genType`. The root entry, `xote/client`, and the core client subpaths expose typed signals, computeds, effects, props, view nodes, and HTML helpers.

Import the same runtime modules from TypeScript:

```ts
import { Signal, View } from "xote/client";

const message = Signal.make("Hello from TypeScript");

const app = View.element("p", [], [], [
  View.signalText(() => Signal.get(message)),
]);

View.mountById(app, "app");
```

The router, SSR, hydration, and MDX entries can still be imported from JavaScript or TypeScript, but the first generated declarations focus on the browser client surface. For fully typed Xote components and JSX, use ReScript source files and import the compiled JavaScript into the rest of the app.

## Working with Existing Apps

Xote can be mounted into any existing DOM node:

```html
<div id="xote-widget"></div>
<script type="module" src="/src/widget.js"></script>
```

```js
import { Signal, View } from "xote/client";

const label = Signal.make("Save");

View.mountById(
  View.element("button", [], [["click", () => Signal.set(label, "Saved")]], [
    View.signalText(() => Signal.get(label)),
  ]),
  "xote-widget",
);
```

That makes JavaScript and TypeScript usage a good fit for small widgets, gradual adoption, and integration tests around the compiled runtime. New full applications should still prefer ReScript so the JSX and type system work together.
